# Falsification Conditions

Status: Draft  
Purpose: Define what would count as serious failure.

---

## 1. Formal Failure

The formal core fails or requires revision if:

- antisymmetry is not preserved under declared admissible transformations
- d₂d₁=0 is not true under the chosen cochain definitions
- triality action is not well-defined
- P𝒯 is not a projection
- 𝒟 becomes empty except for trivial zero when non-trivial states are required
- Rep(Δ) cannot be distinguished from arbitrary assertion

---

## 2. Ontological Failure

The ontology fails if:

- Φ becomes an agent
- Vortex becomes a selector
- K becomes an optimization target
- κ becomes numeric threshold
- QE becomes refusal policy
- audit becomes authority
- memory influences Φ
- downstream layers define root ontology

---

## 3. Implementation Failure

Implementation fails if:

- code contains objective optimization over Φ
- code ranks candidate trajectories as preferred
- audit writes to field state
- memory alters Vortex dynamics
- projection feeds back into Φ
- ASIMULATOR or ASI_MOD can run as standalone valid architecture
- CI permits forbidden reverse dependencies

---

## 4. Empirical Failure

Safety-relevant claims weaken if:

- destructive dominance patterns pass all approximations as admissible
- red-team prompts cause the dialogic layer to claim ontological truth
- QE marker is converted into content
- Rep approximation silently becomes runtime policy
- users reliably misunderstand VECTAETOS as a decision system despite documentation

---

## 5. Communication Failure

Public presentation fails if readers infer:

- VECTAETOS is a chatbot
- Epistemic Cryptography encrypts data
- QE is moral refusal
- κ is tunable safety threshold
- triality alone guarantees safety
- ZMYSEL computes semantic meaning

---

## 6. Scientific Failure

Academic credibility fails if:

- claims exceed proofs
- metaphors replace definitions
- no explicit examples of admissible Δ are provided
- Rep(Δ) remains undefined without being marked primitive
- no implementation audit is possible
- no red-team protocol exists

---

## 7. Recovery Rule

If any failure occurs:

1. Do not hide it.
2. Classify it.
3. Update claim register.
4. Lower public claim strength.
5. Patch formal definition or implementation.
6. Record version boundary if ontology changes.
